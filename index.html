JIT と OPcache を
「正しく」期待する

― PHP 高速化の効きどころ ―

ma_me / PHP Conference Japan 2026

「JIT を入れれば速くなる」は半分しか正しくない。

何が速くなり、何が速くならないのかを
計測で見極める。

目次

    1. イントロ / 前提合わせ

    このセッションのゴール

    • JIT / OPcache がそれぞれ「どこに効くのか」を、雰囲気ではなく計測で理解する
    • 「JIT 入れたのに速くならない」の正体を理解して帰ってもらう

    PHP 実行の流れ

    PHP ソース → 字句 / 構文解析 → OPCODE（中間表現）→ Zend VM が実行

    • OPcache … 「ソース → OPCODE」の変換結果をキャッシュ＝コンパイル段階に効く
    • JIT … OPCODE の実行段階を速くする

    OPcache とは

    • PHP は毎リクエスト、コードを OPCODE（中間命令）へコンパイルしている
    • OPcache はそのコンパイル結果をメモリにキャッシュして使い回す
    • ＝コンパイルコストの削減がメイン。実行そのものを速くするわけではない

    JIT とは（PHP 8.0+）

    • 通常 OPCODE は Zend VM が 1 命令ずつ解釈実行している
    • JIT は OPCODE を実行時に ネイティブ機械語へ変換し CPU で直接実行
    • OPcache の上で動く＝OPcache が無効なら JIT も効かない

    OPcache・JIT を入れるとどこまで速くなるのか。
    作って・測って・確かめてみよう。

    2. OPcache が効くケース

    OPcache の効果はコード量に比例

    効果は コード量（コンパイル対象の規模）にほぼ比例する。

    ▽薄い自作スクリプト … コンパイルコストが元々小さい → 効果も薄い

    △大量のクラス / ファイルを読む … 省略が積み上がる → 大きい

    一番効く例：Laravel（フレームワーク）

    Laravel の初期 Welcome ページを OPcache OFF / ON で別プロセス 30 回起動して比較（1 リクエストのブート時間）。

    構成1 リクエストのブート比
    OPcache OFF146.89 ms1.0×（基準）
    OPcache ON71.89 ms約 2.04× 速い（約 75ms 短縮）

    実測 / Docker・PHP 8.5。計測コード: bench/laravel_request.php / bench/laravel_boot_bench.sh

    逆効果パターン：薄いスクリプトの寄せ集め

    関数 1 個だけの極小ファイルを N 個 require するだけ。やっていることはこれだけ：

    
// ① 関数1個だけの極小ファイルを N 個生成
for ($i = 0; $i < $n; $i++) {
    file_put_contents("thin_{$i}.php", "<?php function thin_fn_{$i}(\$x){ return \$x + {$i}; }");
}
// ② それを require するだけ＝「薄いスクリプトの寄せ集め」
$t0 = hrtime(true);
for ($i = 0; $i < $n; $i++) {
    require "thin_{$i}.php";   // OFF: 毎回コンパイル / ON: file_cache から読むだけ
}
echo (hrtime(true) - $t0) / 1e6, " ms\n";

    薄いスクリプト：ON でも速くならない

    require ファイル数OPcache OFFOPcache ON比
    1 個（＝薄いスクリプト）0.030 ms0.036 ms0.83×（速くならない）
    50 個0.199 ms0.243 ms0.82×
    500 個1.636 ms1.891 ms0.87×
    • どの行も ON は速くならない。むしろ僅かに遅い
    • キャッシュから読むオーバーヘッドが、薄いコードをそのまま読むコストを上回った
    • 効くのはファイル「数」ではなくコンパイル「総量」（＝コード量）

    OPcache は「実行」ではなく「毎回のコンパイル」を省く。

    コード量が多いフレームワークが
    最大の受益者。

    3. 計測の枠組み

    計測の枠組み

    • ツール：PHPBench（@Revs＝内側ループ回数 / @Iterations＝反復）
    • 比較軸：JIT OFF vs ON、さらに mode（function vs tracing）
    • 原則：JIT が速くするのは OPCODE の実行部分。標準関数や IO の中身（ネイティブ C）は速くできない

    3-1. 最大の落とし穴：JIT が動いていない

    • PHP 8 以降 opcache.jit の既定は tracing … だが opcache.jit_buffer_size の既定が 0
    • バッファが 0 だと mode を何にしても JIT は動かない
    • CLI（PHPBench）は opcache.enable_cli=1 が別途必要（FPM とは別 ini）
    
opcache.enable=1
opcache.enable_cli=1
opcache.jit=tracing
opcache.jit_buffer_size=128M   ; ← これを明示しないと JIT は無効

    3-1. opcache.jit は 4 桁 CRTO

    桁意味主な値
    C (CPU)CPU 固有最適化0=なし / 1=AVX
    R (Register)レジスタ割り付け0=なし / 1=局所 / 2=全域
    T (Trigger)いつ compile するか0=ロード時 / 1=初回 / 3=ホット関数 / 5=トレーシング
    O (Optimization)最適化レベル1=最小 … 5=手続き内解析

    function＝1205（T=0）/ tracing＝1254（T=5）。本質は第 3 桁（トリガ）の違いだけ。

    3-2. function JIT と tracing JIT

    function（T=0）

    • 関数をまるごと機械語化
    • ホットかどうかに関係なく対象
    • ウォームアップ不要

    tracing（T=5）

    • ホットな実行経路（trace）だけ compile
    • 関数境界を越えてインライン化・実型で特殊化
    • ウォームアップ必要

    3-2. どちらを選ぶか（損益分岐）

    観点function (T=0)tracing (T=5)
    compile タイミングロード時に全関数実行を観測しホットな trace を後から
    ウォームアップ不要（最初から効く）必要（検出までは素の VM）
    最適化の質関数単位どまり境界を越えて特殊化
    得意起動直後・短命処理ホットループ・常駐ワーカー

    短命 CLI・1 回こっきりの処理では tracing の warmup を回収できない。「常に tracing が最速」ではない。

    4. JIT はどこで効くのか

    4-1. ケース①：四則演算 / bcmath

    • 四則演算・ループ・文字連結 ＝ OPCODE そのもの（ADD / MUL …）→ JIT がネイティブ命令に
    • ネイティブ float 演算は JIT ON で大きく改善（OPCODE が支配的）
    • bcmath は内部 C 実装 → JIT の恩恵は限定的

    計算が OPCODE で表現される処理ほど JIT が効く

    4-2. ケース②：コールバックを取る標準関数

    • 標準関数でも、コールバック部分はユーザー定義 ＝ OPCODEが発生
    • array_map / usort … 中身のループは C だが、各要素のクロージャは JIT 対象
    • コールバックの処理が重いほど JIT の効果が見える

    教訓：標準関数でも「ユーザーコードを実行する隙間」があれば JIT は効く。

    4-3. ケース③：もう一つの JIT ＝ PCRE JIT

    • PHP の JIT とは別系統、PCRE ライブラリ自身が持つ JIT（pcre.jit）
    • 正規表現エンジンが内部でパターンを機械語化する仕組み
    • 計測は PCRE JIT ON / OFF で比較

    「JIT」と一口に言っても複数あり、効くレイヤーが違う。混同しないこと。

    ここからが実践的な核。
    型付きの値オブジェクトで計算を表現する層に JIT が効く。

    4-4. 題材：Money 値オブジェクト

    型付き（readonly + 型宣言）

    
final class Money
{
    public function __construct(
        private readonly int $amount
    ) {}

    public function add(Money $o): self
    { return new self($this->amount + $o->amount); }

    public function multiply(int $f): self
    { return new self($this->amount * $f); }
}

    型なし（型宣言を外しただけ）

    
final class MoneyLoose
{
    private $amount;
    public function __construct($amount)
    { $this->amount = $amount; }

    public function add($o)
    { return new self($this->amount + $o->amount); }

    public function multiply($f)
    { return new self($this->amount * $f); }
}

    4-4. 計測マトリクス（tracing JIT）

    水準内容JIT ON での読み筋
    A: 生スカラー演算$sum + $a; $sum * 2; をループ大きく改善（上限の目安）
    B: 型付き Moneyadd / multiply をループ大きく改善し A に肉薄
    C: 型なし MoneyLooseB と同等処理改善は鈍い（型ガードが残る）
    • JIT ON で B（型付き）が A に肉薄：型が静的に分かり、メソッドをインライン化・プロパティを直接アクセス・int 演算に特殊化
    • C（型なし）は型不明ゆえガード・汎用パスが残り改善が鈍い

    B と C の差（＝型を付けたかどうか）が JIT ON で開く。

    設計上の正しさ（型安全）が、
    そのまま JIT の効きに直結する。

    4-4. 補足（誠実に / 質疑対策）

    • イミュータブルは毎演算で new → アロケーション / GC コストは JIT でも消えない。A には完全一致しない
    • 効くのは tracing JIT ＋ ホットループが前提。単発呼び出しでは出ない

    4-4. 代入の計測（コンストラクタ代入）

    
final class TypedPoint {
    public function __construct(
        private readonly int $x,
        private readonly int $y,
    ) {}
}
final class LoosePoint {
    private $x; private $y;
    public function __construct($x, $y) { $this->x = $x; $this->y = $y; }
}
    • JIT ON で TypedPoint は代入時の型検証が省かれ、int は refcount も不要で素のストアに
    • new のアロケーションは両者共通かつ JIT でも消えない。差は代入パスぶんに限られる

    4-4. 比較演算子：== vs ===

    水準比較JIT ON での読み筋
    D1int === int1 命令の整数比較に特殊化
    D2int == intD1 と同じ命令に潰れ差が消える
    D3int == 数値文字列突出して遅い・JIT でも縮まらない（強制変換は C 側）
    D4int === string型不一致で即 false、安い

    JIT は型が揃えば == / === の差を消す。差が残るのは coercion のときだけ。

    == の正しさの落とし穴（0 == "a" 等）は性能とは別軸。混ぜない。

    4-4. ドメイン層への接続

    • ドメイン層を「型付きの値オブジェクト＋計算」で構成すると、JIT の恩恵を最大化
    • 逆に、メソッドの中身が標準関数や IO の寄せ集めだと効かない（中身は C で JIT 対象外）

    「クラスで包んだ計算」に効く
    のであって「クラスだから効く」ではない

    4-5. ケース④：mode 自体で結果が変わる

    同じ「JIT ON」でも mode で差が出る。アロケーションを排した純粋比較：生成済みインスタンスのプロパティをホットループから読んで加算するだけ。

    
final class TypedBox {
    public function __construct(private readonly int $value) {}
    public function value(): int { return $this->value; }
}
// HotLoopBench:  N=10000 をループして value() を毎回呼ぶ
public function benchTyped(): void
{ $s = 0; $b = $this->typed; for ($i=0;$i<self::N;$i++) $s += $b->value(); }

    4-5. 計測結果（mode＝平均 1 回あたり）

    水準JIT OFFfunctiontracingOFF→tracing
    A: 生スカラー17.74μs4.91μs3.77μs約 4.7倍
    B: 型付き TypedBox91.64μs49.66μs38.45μs約 2.4倍
    C: 型なし LooseBox92.62μs52.09μs38.04μs約 2.4倍
    • OFF → function → tracing で単調に速くなる。同じ「JIT ON」でも function→tracing でさらに 2〜3 割縮む
    • tracing が速い理由：ホットループ内の value() を trace にインライン化できるから

    4-5. 誠実な注意（重要）

    • このループでは 型付き B と型なし C の差はほぼ消える（tracing で 38.4 vs 38.0μs）
    • tracing は観測した実行時の型で特殊化するため、int が流れていれば宣言の有無は効きにくい
    • 型宣言が効くのは ConstructBench のような「代入時の型検証」パス。steady-state の読み出しでは差が出にくい

    「型を付ければ JIT で速くなる」を単純化しすぎない。

    「JIT を入れた / 入れない」は粗すぎる。

    mode（function / tracing）まで含めて測る。

    5. まとめ ― レイヤーで考える

    5-1. 役割分担の整理

    仕組み効く対象典型的な場所
    OPcacheコード量が多い処理（コンパイルコスト）外部依存・フレームワーク層
    JITOPCODE 計算が支配的な処理コアドメイン・数値計算層

    5-2. アーキテクチャとの相乗効果

    • 外部依存の層（フレームワーク・I/O） → OPcache が効く
    • コアドメインの層（純粋なロジック・計算） → JIT が効く
    • 「関心の分離」をすると、高速化の効きどころも自然と分離される

    5-3. 持ち帰ってほしいこと

    • JIT は「OPCODE の実行」を速くするだけ。標準関数の中身は速くならない
    • JIT が効くのは数値計算・コールバック・ドメインロジック
    • OPcache はコード量に比例して効く（フレームワーク向き）
    • 「JIT」には PCRE JIT など別系統もある
    • レイヤー設計とキャッシュ / JIT の効きどころは対応している
