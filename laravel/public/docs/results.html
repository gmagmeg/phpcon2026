→ 同一の welcome リクエスト（boot で included 407 / declared 446、full request で included 523 / declared 514）を、OPcache OFF（毎プロセスで全ファイルを再コンパイル）と ON＋opcache.file_cache（コンパイル済み opcode をプロセス跨ぎで再利用＝FPM 相当）で、別プロセス 30 回起動して平均。OPcache が省くのは「実行」ではなく「毎リクエストのコンパイル」なので、コンパイル支配的な boot で約 2.04×、ランタイム処理（描画）を含む full request では 約 1.78× に縮む——“効くのはコンパイル部分”が比の差として出ている。JIT は本計測の対象外として両条件とも無効固定。 bench/laravel_request.php / bench/laravel_boot_bench.sh

計測環境: PHP 8.5（Docker・Debian bookworm）。他コンテナを停止したアイドル状態での実測で、絶対値・比とも他負荷の有無でほぼ不変（再現性あり）。再現: docker compose exec app sh bench/laravel_boot_bench.sh 30 boot（request でフルリクエスト）。

→ 10,000要素の末尾を探索。in_array は線形 O(n)、isset/array_key_exists はハッシュ参照 O(1)。OPcache/JIT の ON・OFF で in_array はほぼ不変（11.8μs↔11.9μs・約1%差で誤差内）。いずれも内部関数／専用 opcode のため、差は完全に計算量由来で OPcache・JIT は無関係。 計測環境: PHP 8.5.7 / xdebug ❌（ON時 opcache ✔ / OFF時 opcache ❌）。再計測 2026-06-22。 benchmarks/InArrayBench.php

isset / array_key_exists は1回が ~20〜100ns と極小のため rstdev が大きく（±4〜6%）、mode 推定もブレる（ON/OFF の前後が逆転しうる）。絶対値の僅差より、桁（in_array 比 約150倍 / 約110倍）で読むのが正しい。 ON: 既定設定 / OFF: --php-config='opcache.enable_cli: 0'

総実行時間の目安（revs × its = 2000 × 5 = 10,000回分・mode×回数の概算）— ON / OFF: in_array 約117.5ms / 約118.8ms ・ isset 約0.79ms / 約0.22ms ・ array_key_exists 約1.08ms / 約0.31ms。in_array は ON/OFF ほぼ同等（内部関数）。

③ コールバックを取る標準関数（array_map / usort）— JITは効くか（PHPBench・mode）

処理JIT ONJIT OFFJIT効果
手書き foreach（完全 userland）69.480μs92.276μs約1.33倍速（~33%）
array_map（コールバック userland）131.720μs156.356μs約1.19倍速（~19%）
usort（比較コールバックを多数回）2.501ms2.854ms約1.14倍速（~14%）

→ 標準関数でも、渡すコールバックは userland PHP opcode なので JIT が効く。ただし純粋な手書きループ（33%）より効果は小さい（19% / 14%）。これは実行時間の一部が C実装側（配列の反復・ソートアルゴリズム）に費やされ、そこは JIT 非対象だから。「userland 比率が高いほど JIT が効く」が綺麗に出ている。 benchmarks/CallbackBench.php ／ 解説は JITのしくみ

⑥ アロケーションを排した純粋比較（HotLoopBench）— 型付き / 型なし × 3 mode

水準JIT OFFfunction (1205)tracing (1254)OFF→tracing
A: 生スカラー（プロパティ直読み・呼び出しなし）28.617μs7.755μs6.468μs約4.4倍速
B: 型付き TypedBox（value() をホットループ）149.190μs83.457μs55.103μs約2.7倍速
C: 型なし LooseBox（B と同等処理）149.936μs85.713μs55.187μs約2.7倍速

→ N=10000・@Revs(100) / インスタンスは setUp で 1 回だけ生成（毎反復のアロケーションを排除）。読み筋は 2 つ:
(1) 同じ JIT ON でも function → tracing でさらに 2〜3 割速くなる（B: 83.5μs → 55.1μs）。tracing は 関数境界を越えてホットな value() をインライン化できるため。
(2) 型付き B と 型なし C の差はほぼ消える（tracing で 55.1μs vs 55.2μs）。tracing は 観測した実行時型で特殊化するので、int が流れていれば宣言の有無は steady-state では効きにくい。「型を付ければ JIT で速い」の単純化への正直な反例。
A と B/C の差（6.5μs vs 55μs）はメソッド呼び出しフレームのコスト分。 benchmarks/HotLoopBench.php ／ 計測日 2026-06-27 / PHP 8.5.7・PHPBench 1.7 mode

再現: docker compose exec app vendor/bin/phpbench run benchmarks/HotLoopBench.php --report=aggregate（tracing）／ function: --php-config='opcache.jit: function' ／ OFF: --php-config='opcache.jit_buffer_size: 0'。

⑦ 実践例：イベントソーシング型の入出金（TransferBench）— 連想配列 vs 型付き VO × 3 mode

履歴の表現JIT OFFfunction (1205)tracing (1254)OFF→tracingmem_peak
連想配列（$history[] = [...]）543.583μs460.089μs449.949μs約1.21倍速8.57mb
型付き VO（Account / TransactionEvent）1.122ms907.395μs737.002μs約1.52倍速6.05mb

→ DB から SELECT した体裁の連想配列（日時・操作・金額・名称）5,000 件を入力に、1 件ずつ「出金者の合計から引いて出金履歴を記録 / 入金者の合計に足して入金履歴を記録」を回す。履歴の表現だけを 2 通りで実装して比較。読み筋は 3 つ:
(1) 絶対値は連想配列の方が速い（tracing 450μs vs 737μs）。型付き VO は 1 件ごとに new TransactionEvent（アロケーション）＋メソッド呼び出しのコストが乗るため。
(2) JIT の改善率は型付き VO の方が大きい（OFF→tracing で 1.52倍 vs 1.21倍）。メソッド呼び出し・型付き演算が userland OPCODE として JIT の対象になるから。とくに function（907μs）→ tracing（737μs）の差が大きい＝ tracing が withdraw/deposit をインライン化できる。連想配列版は配列追記（zend_hash＝C実装）が支配的で function↔tracing がほぼ同じ。
(3) メモリは型付き VO の方が小さい（6.05mb vs 8.57mb）。連想配列は 1 イベントごとにキー文字列＋ハッシュ構造を持つため太る。 benchmarks/TransferBench.php / Account.php / TransactionEvent.php ／ 計測日 2026-06-28 / PHP 8.5.7・PHPBench 1.7 mode

再現: docker compose exec app vendor/bin/phpbench run benchmarks/TransferBench.php --report=aggregate（tracing）／ function: --php-config='opcache.jit: function' ／ OFF: --php-config='opcache.jit_buffer_size: 0'。
