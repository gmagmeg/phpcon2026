→ 10,000要素の末尾を探索。in_array は線形 O(n)、isset/array_key_exists はハッシュ参照 O(1)。OPcache/JIT の ON・OFF で in_array はほぼ不変（14.8μs↔14.8μs）。いずれも内部関数のため、差は完全に計算量由来で OPcache・JIT は無関係。 計測環境: PHP 8.5.7 / xdebug ❌（ON時 opcache ✔ / OFF時 opcache ❌）。 benchmarks/InArrayBench.php

isset / array_key_exists は1回が ~10〜30ns と極小のため rstdev が大きい（±8〜11%）。絶対値の僅差より、桁（約920倍 / 約490倍）で読むのが正しい。 ON: 既定設定 / OFF: --php-config='opcache.enable_cli: 0'

総実行時間（revs × its = 2000 × 5 = 10,000回分、sum(result_time_net)）— ON / OFF: in_array 147.0ms / 157.4ms ・ isset 0.16ms / 0.28ms ・ array_key_exists 0.31ms / 0.37ms。in_array は ON/OFF ほぼ同等（内部関数）。

ネイティブ float 演算子 vs bcmath（PHPBench・mode）

処理JIT ONJIT OFFJIT効果
ネイティブ * +（float）1.253μs4.562μs約3.6倍速
bcmath（bcmul/bcadd）260.511μs274.366μsほぼ差なし

→ 同じ「掛けて足す」を1000回繰り返すループ。ネイティブ演算子は言語opcodeなので JIT が効く（1.25μs↔4.56μs）。一方 bcmath は C実装の内部関数なので JIT 非対象（261μs↔274μs で大差なし）。しかも文字列ベースの任意精度演算ゆえ、ネイティブの約200倍遅い。「JITが効く境界＝opcode か C内部関数か」を最も鮮明に示す対比。 benchmarks/BcmathBench.php ／ 解説は JITのしくみ

総実行時間（revs × its = 200 × 5 = 1,000回分、sum(result_time_net)）— JIT ON / OFF: ネイティブ 1.30ms / 4.54ms（JITで約3.5倍）・ bcmath 275.3ms / 267.6ms（差なし）。

③ コールバックを取る標準関数（array_map / usort）— JITは効くか（PHPBench・mode）

処理JIT ONJIT OFFJIT効果
手書き foreach（完全 userland）69.480μs92.276μs約1.33倍速（~33%）
array_map（コールバック userland）131.720μs156.356μs約1.19倍速（~19%）
usort（比較コールバックを多数回）2.501ms2.854ms約1.14倍速（~14%）

→ 標準関数でも、渡すコールバックは userland PHP opcode なので JIT が効く。ただし純粋な手書きループ（33%）より効果は小さい（19% / 14%）。これは実行時間の一部が C実装側（配列の反復・ソートアルゴリズム）に費やされ、そこは JIT 非対象だから。「userland 比率が高いほど JIT が効く」が綺麗に出ている。 benchmarks/CallbackBench.php ／ 解説は JITのしくみ
