逆効果パターン

薄いスクリプトの寄せ集め
関数 1 個だけの極小ファイルを N 個 require する


/**
 * 関数1個だけの極小ファイルを N 個生成
 */
for ($i = 0; $i < $n; $i++) {
    file_put_contents("thin_{$i}.php", /* 関数1個だけの PHP コード */);
}

// 読み込み
$t0 = hrtime(true);
for ($i = 0; $i < $n; $i++) {
    require "thin_{$i}.php";   // OFF: 毎回コンパイル / ON: file_cache から読むだけ

echo (hrtime(true) - $t0) / 1e6, " ms\n";

2. JIT の効果計測

JIT の概要

  • OPcache 前提
  • キャッシュした OPCODE を
    ネイティブ機械語    に変換して実行
  • Zend VM を介さず CPU が直接走るため
    計算ヘビーな処理ほど効きやすい

JIT の計測環境

CPU / アーキネイティブ x86_64
（GitHub Actions ubuntu-latest）
PHP8.5（OPcache 前提）
JIT モードOFF / function / tracing
（jit_buffer_size = 0 / 64M）
計測ライブラリPHPBench

実は３つある JIT の実装

CPU アーキテクチャ ごとにコード生成器が分かれている

  • x86（32-bit / i386）
  • x86_64（今回使用）（64-bit / AMD64）
  • ARM64（AArch64 / Apple Silicon 等）

x86_64 の JIT が安定。
ARM64 は比較的新しく、同じコードでも速度が異なる。
→ 後ほど実例を紹介

JIT が効く箇所

JIT は標準関数に
どこまで作用するのか計測する

isset と in_array() 速度差は縮まるのか

in_array：先頭から走査する 線形探索 O(n)

isset：キー有無を引く ハッシュ参照 O(1)

10,000 要素の末尾を探索し、OPcache/JIT の ON と OFF で比較 


				// in_array：値の中から探す（得意：値検索）
				$values = range(1, 10000);
				in_array(10000, $values, true);    // 末尾までフルスキャン
			
			

// isset：キーの有無を引く（得意：キー参照）
$keys = array_fill_keys(range(1, 10000), true);
isset($keys[10000]);               // 1 回のハッシュ参照

どの関数も変わらず JIT ありの方がむしろ僅かに 遅い

型定義、型情報にも JIT は効く

型宣言そのものではなく、型付きのユーザー定義コードが最適化対象

型が明確だと、JIT は値の型を推論しやすくなる

Money クラスを定義して計測


final class Money {
    public function __construct( private readonly int $amount) {}
    public function add(Money $o): self { return new self($this->amount + $o->amount); }
    public function multiply(int $f): self { return new self($this->amount * $f); }
}

new には JIT が効かないため、この差が生まれる

new の裏側では C によるメモリ操作が行われる
しかし JIT はこの C の部分までは届かない

より実践的なコードでその効きの差を検証
お題は、クーポン割引・送料計算

題材：クーポン割引・送料計算

  • 1

    小計 カート明細（単価 × 数量）を合計する

  • 2

    割引 適用クーポンで割引額を算出する（定額 / 率 / 条件付き）

  • 3

    送料 配送地域 × 重量から送料を算出する（無料ライン判定あり）

  • 4

    確定 小計 − 割引 ＋ 送料 で請求額を確定する

ベースの実装概要

ミュータブル版は new せず、代入したオブジェクトを返す


// クーポンは interface ＋ 3 実装（定額 / 率 / 条件付き）
interface Coupon { function discountFor(Money $s): Money; }

// 小計 → 割引 → 送料 → 確定（Money は演算のたびに new）
$discount   = $cart->coupon->discountFor($subtotal); // ← new
$discounted = $subtotal->subtract($discount);        // ← new
$total      = $discounted->add($shipping);           // ← new

閑話休題

JIT は ON にした方がいいのか？
ON にした方がいい。
効果がないケースはあれど、遅延するケースはない。
設定値も編集する必要はない。
ただしmsの世界なので、過大な期待は禁物。
ここからは OPcache・JIT 適用時のハマりポイント

設定値で気をつけること

Opcacheのデフォルト設定

アプリケーションとCLIの設定値は別れているので注意。

・Web アプリケーション：Opcache は初期設定で ON

・CLI実行：Opcache は初期設定で OFF
このような実行形式には無効になっている

php artisan queue:work

JIT はデフォルトで OFF になっている

項目8.3 まで8.4 以降
opcache.jittracingdisable
opcache.jit_buffer_size064M
OFF になる理由buffer_size = 0jit = disable

8.4 以降は opcache.jit_buffer_size だけでは JIT は有効化されない

JIT ON ではデバッグしにくい

Xdebug と PHP JIT は併用できない
Xdebug を読み込むと警告が出て、JIT は無効になる

ソースを再読み込み（not ブラウザリロード）しないと、修正内容が反映されない

開発環境では OFF、本番環境では ON にするのが推奨パターン
