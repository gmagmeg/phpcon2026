2. JIT の効果計測
― 意外な結果から

JIT の概要

  • OPcache 前提で、キャッシュした OPCODE をネイティブ機械語に変換して実行
  • Zend VM を介さず CPU が直接走るため、計算ヘビーな処理ほど効きやすい

CPU / アーキネイティブ x86_64（AMD EPYC 7763 / GitHub Actions ubuntu-latest）
PHP8.5（OPcache 前提）
JIT モードOFF / function / tracing の 3 通りで比較（jit_buffer_size = 0 / 64M）
計測ライブラリPHPBench

TODO：全計測の基準アーキを x86_64 に統一する（現状 Money / カートの計測は ARM64。x86_64 で再計測し、アーキ差は第3章末の「おまけ」でのみ扱う）

JIT はアーキごとに別実装

PHP の JIT（DynASM ベース）は、アーキごとにコード生成器が分かれている：

  • x86（32-bit / i386）
  • x86_64(64-bit / AMD64）
  • ARM64（AArch64 / Apple Silicon 等）

ARM64 対応は比較的新しく、同じコードでも速度が異なる。→ 後ほど実例を紹介

JIT が効くのはここ

どちらも内部関数。JIT で速くなる？

3. 実践編
― 時間の在処を OPCODE 側に寄せる

型が安定した「計算」には効く

  • 型付きのドメインロジック＝OPCODE 中心の計算 → JIT の得意分野
  • ただしオブジェクト生成（new）は C 側の alloc → そこには届かない

「クラスで包んだ計算」に効く
のであって「クラスだから効く」ではない

JIT の 2 つの mode：function と tracing

  • function：関数まるごとを機械語化する
  • tracing（既定）：実行を観測し、ホットな経路だけを機械語化する

同じ「JIT ON」でも mode で結果が変わる

型があるほど JIT が最適化しやすい

引数・戻り値の型宣言などで型が安定しているほど、JIT は次の最適化をしやすい

  • 型チェック・ガードを省略しやすい
  • int + int を 整数加算として特殊化しやすい
  • プロパティアクセスを より直接的な経路へ最適化しやすい
  • 小さなメソッドを インライン展開しやすい

Money オブジェクト

readonly + 型宣言
引数・戻り値・プロパティすべてに型がある。


final class Money
{
    public function __construct(
        private readonly int $amount
    ) {}

    public function add(Money $o): self
    { return new self($this->amount + $o->amount); }

    public function multiply(int $f): self
    { return new self($this->amount * $f); }
}

計測内容：オブジェクト生成をループ外へ出せるか（new の位置に注目）

手続き


for ($i = 0; $i < 1000; $i++) {
    $acc += ($i + $i) * 2;
}

ミュータブル（new はループ外で1度だけ・中は破壊的更新）


$m = new MutableMoney(0);                     // ← ループ外
for ($i = 0; $i < 1000; $i++) {
    $m->set($i)->add($m)->multiply(2);        // in-place（new なし）
    $acc += $m->amount();
}

イミュータブル（add / multiply が毎回 new する）


for ($i = 0; $i < 1000; $i++) {
    $m = new Money($i);                        // ← ループ外に出せない
    $acc += $m->add($m)->multiply(2)->amount();
}

題材：クーポン割引・送料計算

EC のカート確定処理を、2 実装（連想配列版 / 型付き VO 版）で計測する

  • 1

    小計 カート明細（単価 × 数量）を合計する

  • 2

    割引 適用クーポンで割引額を算出する（定額 / 率 / 条件付き）

  • 3

    送料 配送地域 × 重量から送料を算出する（無料ライン判定あり）

  • 4

    確定 小計 − 割引 ＋ 送料 で請求額を確定する

実装：クーポンはポリモーフィズム


interface Coupon { public function discountFor(Money $subtotal): Money; }
final class FixedCoupon     implements Coupon { /* 小計を超えない定額   */ }
final class RateCoupon      implements Coupon { /* 小計の N%           */ }
final class ThresholdCoupon implements Coupon { /* 閾値以上で固定額     */ }

// 型付き VO 版：小計 → 割引 → 送料 → 確定
$subtotal = Money::zero();
foreach ($cart->lines as $line) $subtotal = $subtotal->add($line->subtotal());
$discount   = $cart->coupon->discountFor($subtotal);   // ← 多態 dispatch
$discounted = $subtotal->subtract($discount);
$shipping   = $policy->feeFor($discounted, $region, $weight);  // 地域×重量
$total      = $discounted->add($shipping);

配列版は同じルールを switch ($spec['type']) の手続き分岐で実装（CartCheckoutArrayBench）。VO 版は上記クラス群（CartCheckoutTypedBench）。5,000 カート × 4 明細、両版で結果一致を検証済み。

計測結果：alloc を削ると JIT が効く

実装JIT OFFfunctiontracingOFF→tracing
配列1.60ms1.03ms0.94ms約 1.70 倍速

配列 = 連想配列版 / VO = 型付き VO 版 / 低alloc = ループ内 new を排した VO（PHP 8.5.7 / ARM64・100 revs × 5 it）

VO の重さは Money 生成（C 側 alloc）。ループ内の new を削ると JIT の効き幅が 1.37× → 2.04× に化け、tracing で配列に迫る。

効き幅の差：バイトコード vs C 側

  • 配列版は int 演算＋配列アクセス主体 → JIT がバイトコードを特殊化 → 約 1.70 倍速
  • VO 版は new Money() のオブジェクト生成（C 側 alloc）が残る → JIT が届かず効きは薄い（約 1.37 倍）
  • 多態 discountFor() も 3 型混在で virtual call が残りインライン化されにくい
  • 参考：純粋な整数ループなら tracing で 約 9 倍速（JIT が最も効く形）
  • 結論：JIT はバイトコードを速くする。VO の生成コストは C 側に残り、効き幅も絶対値も差が残る（datetime／Carbon と同じ構図）

値オブジェクトの等価判定：getter 直比較 vs equal()


// 型付き VO（Money = readonly int）
$priceA = new Money(12345);
$priceB = new Money(12345); // 同額・別インスタンス

// 同じ等価判定を 2 通りで書く
$priceA->amount() === $priceB->amount(); // D1 getter 直比較
$priceA->equal($priceB);                 // D2 equal() メソッド

// Money::equal() の中身は int 同士の ===
public function equal(Money $o): bool {
    return $this->amount === $o->amount;
}

水準比較JIT OFFfunctiontracing
D1getter 直比較 ===54ns27ns26ns

JIT ありならそれほど大きな差は出ず。
面白いのはgetter()同士の比較よりも引数渡しの方が早かったこと
getter()を２呼ぶ回数分、コストはかかる

値オブジェクトの等価判定：総括

  • getter 直比較も equal() メソッドも、JIT 後は 同じ ~26ns に潰れる（同じ命令に特殊化）
  • 型付き VO 同士なら equal() の呼び出しを JIT がインライン化 ＝ カプセル化のコストはゼロ
  • JIT が無くても equal() は不利にならない（getter 2 回より呼び出し 1 回が安い）

「速いから getter を晒す」のではなく
「VO に equal() を持たせる」設計を選んでよい

実践編のまとめ

  • 型付きの計算ロジックには効く。効き幅を決めるのはループ内の new の数（C 側 alloc）
  • ポリモーフィズム・equal() のコストは JIT 後ほぼゼロ → 設計を速度のために曲げる必要はない
  • メソッドの中身が標準関数や IO の寄せ集めだと効かない（中身は C で JIT 対象外）

4. まとめ ― 効きどころを予測する

4-1. 役割分担の整理

仕組み効く対象典型的な場所
OPcacheコード量が多い処理（コンパイルコスト）外部依存・フレームワーク層
JITOPCODE 計算が支配的な処理コアドメイン・数値計算層

4-2. アーキテクチャとの相乗効果

  • 外部依存の層（フレームワーク・I/O） → OPcache が効く
  • コアドメインの層（純粋なロジック・計算） → JIT が効く
  • 「関心の分離」をすると、高速化の効きどころも自然と分離される

4-3. 実アプリでは何%効くのか

原則からの予測：Web アプリは IO・内部関数・フレームワークが支配的
→ JIT の効きは数%止まりのはず

4-4. 計測の落とし穴 ―「入れたつもり」を防ぐ

  • phpbench の --php-config は JSON 形式で渡す
    （例：--php-config='{"opcache.jit_buffer_size": 0}'）
    → 形式を誤ると意図しない設定のまま計測してしまう
  • opcache.jit_buffer_size = 0 は JIT 無効と同義 ―「ON にしたつもり」の典型

4-6. 持ち帰ってほしいこと

  • JIT が速くするのは OPCODE の実行だけ。C の世界（内部関数・new・PCRE）には届かない
  • だから効く / 効かないは「時間がどこで消えているか」で計測前に予測できる
  • OPcache はコード量に比例して効く（フレームワーク向き）、JIT は計算が支配的な処理に効く
  • 「JIT」には PCRE JIT など別系統もある。アーキ（x86_64 / ARM64）でも結果が変わる
  • 設計（VO・ポリモーフィズム）を速度のために曲げる必要はない。削るならループ内の new
