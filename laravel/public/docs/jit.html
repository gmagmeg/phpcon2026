JITのしくみ — なぜ Math で効いて in_array で効かないのか

JIT が「何をコンパイルするか」で効き方が決まる

一言で

MathBench は PHPで書いたタイトな数値ループ＝インタプリタが opcode を1個ずつ実行する処理。JIT はその opcode 列をネイティブの機械語に変換するので速くなる。

一方 in_array などの内部関数は最初から C で書かれた機械語。JIT が触るのは「あなたのPHPの opcode」であって「組み込み関数の C 実装」ではないので、効かない。

JIT が効く3条件（MathBench は全部満たす）

① ユーザーランドPHPのループであること

MathBench は for の二重ループ（外2000 × 内最大100）で、最大20万回の演算が PHP opcode として回る。JIT はこの opcode 列をまとめて機械語化し、VMのディスパッチ（fetch→decode→execute を1 opcodeごとに繰り返す）オーバーヘッドを消す。
対して in_array は、PHP側は in_array(...) というたった1つの呼び出し opcode。中身の線形探索は C の中なので、JIT がコンパイルできる PHP opcode がほぼ無い＝効きようがない。

② 数値（float）演算が主であること

PHPの変数は zval（型タグ付きの箱）。インタプリタは演算のたびに型チェック＋箱の出し入れ（boxing/unboxing）をする。JIT（特に tracing JIT）は「この変数はずっと double だ」と特定できると、値をCPUレジスタにネイティブの double として置き、生の FPU/SSE 命令を直接発行する。zval の手間が丸ごと消えるので、ここが最大の伸びしろ。

③ ループ中に外部呼び出し・I/Oが無いこと

MathBench の中身は四則演算と分岐だけ。C関数呼び出しもI/Oもないので、JIT化したコードが中断なく走り続けられる。

効くかどうかの本質

「useful work に対するインタプリタ・オーバーヘッドの比率」で決まる。

1 opcode あたりの仕事インタプリタ下駄の比率JIT
数値ループ（MathBench）小（add 1回など）大（下駄が支配的）✅ 効く
内部関数呼び出し（in_array）大（1呼び出しで全走査）小（下駄は誤差）❌ 効かない

数値ループは「1 opcode の仕事が小さいのに、毎回ディスパッチの固定コストを払う」ので下駄が支配的 → JIT がそれを消すと大きく速くなる（実測 409μs → 1.82ms、約4.5倍）。in_array は「1 opcode（=1呼び出し）で1万件走査」と仕事が大きく、下駄は元から誤差 → JIT の出番なし。

最も鮮明な実例: ネイティブ演算子 vs bcmath

同じ「掛けて足す」ループを2方式で実装すると、JITの境界がそのまま出る。

処理JIT ONJIT OFF意味
ネイティブ * +（float）1.253μs4.562μs言語opcode → JITが効く（約3.6倍）
bcmath（bcmul/bcadd）260.511μs274.366μsC実装の内部関数 → JIT非対象（不変）

演算子は opcode なので JIT が機械語化できるが、bcmul() は in_array() と同じ「C実装の組み込み関数」。だから JIT は効かず、さらに文字列ベースの任意精度ゆえネイティブの約200倍遅い。 benchmarks/BcmathBench.php

標準関数でも JIT が効く2パターン

① コールバックを取る標準関数（array_map / usort）

標準関数の本体は C でも、渡すコールバックは userland PHP opcode。だからそこに JIT が効く。

処理JIT ONJIT OFFJIT効果
手書き foreach（完全 userland）69.480μs92.276μs約1.33倍（~33%）
array_map（コールバック）131.720μs156.356μs約1.19倍（~19%）
usort（比較コールバック多数回）2.501ms2.854ms約1.14倍（~14%）

純手書きループ（33%）より効果が小さいのは、時間の一部が C側（配列反復・ソート）に費やされ JIT 非対象だから。「userland 比率が高いほど JIT が効く」がそのまま出ている。 benchmarks/CallbackBench.php

② 正規表現の「もう一つの JIT」= PCRE JIT

preg_* は C(PCRE)実装なので opcache の JIT は効かない。だが PCRE は独自の JITを持ち、ini の pcre.jit（既定ON）で制御する。これは opcache の JIT とは別物。

処理pcre.jit ONpcre.jit OFFopcache.jit OFF
preg_match ×2000511μs4.440ms510μs
preg_match_all ×20001.187ms7.852ms1.205ms

pcre.jit=0 で 約8.7倍 / 約6.6倍遅くなる一方、opcache の JIT を切っても不変（右列）。＝「効いているのは PCRE JIT であって opcache JIT ではない」。PHPには複数の独立した JITがある、という点に注意。 benchmarks/PcreJitBench.php

まとめ

JIT = PHPの opcode を機械語化する仕組み。だから——

  • PHPで書いた計算ループ（MathBench）には効く
  • C実装の組み込み関数（in_array / array_diff / array_unique）には効かない
  • 「速い/遅い」の本質は多くの場合 計算量（O(n) vs O(1)）であり、そこに OPcache も JIT も無関係
