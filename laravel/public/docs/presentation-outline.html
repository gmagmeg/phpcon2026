概要

PHP8.0 で導入された OPcache と JIT は、現在も改善が続いています。 その一方で両方ともデフォルト設定のまま運用しているケースは多々あり、それぞれの特性を活かせていると言い切れる現場も多くないはずです。 例えば計算量について、 in_array より isset のほうが速い、といった断片的な知識は広く知られていますが なぜ isset のほうが速いのか、そこに OPcache や JIT は効くのか、といった踏み込んだ理解はあまり共有されていません。 本セッションでは、こうしたなんとなく知っていることを、どこに、どう効くのかを測って確かめたに置き換えることが目的としています。

話すこと

  • PHP 標準関数や言語機能を計算量で選ぶ視点（in_array / isset / array_unique など）
  • OPcache・JIT の設定がパフォーマンスに与える影響
  • 計測と実際の計測結果

話さないこと

  • OPcache、JITの踏み込んだ内部実装の話
  • JIT コンパイラ自体の理論的背景

構成案（たたき台）

プロポーザルをもとにしたセッションの流れ。各パートはこのリポジトリのベンチ／ドキュメントに紐づけてある（確定ではない）。

#パート狙い・メッセージ裏付け（このリポジトリ）
0 つかみ / 自己紹介 「in_array より isset が速い」——なぜ? そこに OPcache / JIT は効くのか? を問いとして提示。 —
1 前提合わせ：OPcache と JIT は別物 OPcache＝opcodeのキャッシュ、JIT＝opcodeの機械語化。両方デフォルト運用が多い現実。 基本スタック / JITのしくみ
2 計測の土台 「測って確かめる」ための環境・ツール・JIT切替方法を先に共有。再現可能性の担保。 計測ツール / ベンチ & 実行 / レポートの読み方
3 第1部：計算量で選ぶ視点 in_array O(n) vs isset / array_key_exists O(1)。速さの本質は計算量で、ここに OPcache も JIT も無関係。 計測結果 / benchmarks/InArrayBench.php
4 第2部：JITが効くコード 純粋な数値ループ（userland opcode）は機械語化で大きく伸びる（約4.5倍）。 JITのしくみ / benchmarks/MathBench.php
5 第3部：JITが効かないコード C実装の組み込み関数（in_array / array_unique / bcmath）には効かない。形ではなく中身が opcode か C関数かで決まる。 benchmarks/BcmathBench.php / ArrayUniqueBench.php
6 第4部：標準関数でも効く2パターン ① コールバック（array_map / usort）は中身が userland。② 正規表現は PCRE JIT（opcacheのJITとは別物）。PHPには複数のJITがある。 benchmarks/CallbackBench.php / PcreJitBench.php
7 第5部：設定がパフォーマンスに与える影響 opcache.jit / jit_buffer_size / validate_timestamps などの切替で実測がどう動くか。デフォルトのままで良いのか。 README「OPcache / JIT 設定の切替」 / /perf/status
8 まとめ 「速い/遅い」の本質は多くの場合計算量。JIT が効くのは userland opcode のループ。どこに・どう効くかを測って確かめた。 JITのしくみ：まとめ
9 話さないこと（明示） OPcache / JIT の内部実装、JITコンパイラの理論的背景には踏み込まない、と最後に再確認。 —

本セッションの一本柱

「なんとなく知っている」を「どこに・どう効くかを測って確かめた」に置き換える。
軸は3つ：計算量（O(n) vs O(1)）／opcode か C関数か（JITの境界）／設定（デフォルトのままで良いのか）。
